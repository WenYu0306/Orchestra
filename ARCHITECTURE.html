第一层 · 你看到的前端界面 Safari · localhost:5173

技术栈：Vue 3 + Vite，7 个组件拼成单页。Vite 把 /api 开头的请求代理到后端 5000 端口。

页面布局（从上到下）

App.vue 是总控，内嵌 6 个子组件：

TopBar — 最顶部状态栏。绿色圆点=运行中，灰色=就绪，红色=出错。右侧"开始""停止"两个按钮。

ProgressBand — 进度条+计时器。显示"正在评估「XX公司·XX岗位」"之类的实时步骤。

StatCards — 4 个统计卡片：高分（≥80）、中分（60-79）、可试（40-59）、总候选数。

JobList — 职位卡片列表。顶部有"发送选中 N"按钮（indigo 色），勾选卡片后变亮可点。

JobCard — 单张职位卡片。左上角复选框，中间公司名+职位名+分数（大数字），下面是匹配理由和招呼语预览，可复制。

BottomBar — 最底部信息栏。

数据怎么来的

前端不直接调后端 REST API 拿数据。走的是 SSE 实时推送——前端 new EventSource('/api/events') 建立长连接，后端每评完一个岗位就推一条 record 事件过来，前端立刻插一张卡片。整个过程像直播一样，不是先等完再一次性加载。

复选框为什么之前点不动

Safari 的 bug：<label> 包住 <input type="checkbox"> 时，点击 label 会改变 checkbox 的勾选状态，但不触发 Vue 的 @change 事件。所以后端 toggleJob 函数从来没被调到，selectedIds 数组永远是空的，按钮一直是灰的。现在改成了纯 <div> + @click，绕过了这个坑。

第二层 · 后端 API 层 FastAPI · localhost:5000

技术栈：Python 3.12 + FastAPI + uvicorn。入口文件 backend/main.py

7 个 API 端点

GET /api/status 查当前状态：空闲/运行中/验证码/已完成，各层已投了几个。前端 onMounted 时调一次。
POST /api/start 启动全部流程：开浏览器 → 扫码登录 → 搜索 → 评分 → 分层 → 生成招呼语。内部 asyncio.create_task(_run()) 丢到后台跑。
POST /api/stop 设 _stop = True，几秒内任务自然退出。
GET /api/events SSE 长连接。前端 EventSource 挂在这里，实时收到 status / record / complete / error 四种事件。
GET /api/test 不开浏览器，纯测 AI 链路：读简历 → 生成搜索词 → 示例 JD 打分。启动前验证用。
POST /api/send ⚠️ 招呼语发送。当前实现有 bug：导航到职位详情页找沟通按钮，应该改到搜索列表页直接操作。见下方说明。
POST /api/close 手动关 nodriver 浏览器。

关键技术细节

搜索怎么做的：不走 UI 点击。在 nodriver 打开的 Chrome 页面里注入同步 XHR（XMLHttpRequest），直接调 BOSS 的内部 API wapi/zpgeek/search/joblist.json?query=xxx&city=xxx&page=1&pageSize=30。请求从真 Chrome 发出去，cookie 和请求头跟真人一模一样，不会被反爬拦。

详情 JD 怎么拿的：同样方式调 wapi/zpgeek/job/card.json?securityId=xxx，取 zpData.jobCard.postDescription。

"标签"是什么：搜索 API 返回的数据里没有完整 JD，只有字段碎片——职位名、薪资范围、经验要求、学历要求、技能标签、福利列表。拼成一句话扔给 DeepSeek 打快分，叫"标签评分"。分数高的 top 30 才去拿完整 JD 重评。

浏览器为什么不关：之前干跑完 finally 里直接 browser.stop()。现在改成只在 _stop 为 true 时才关——干跑完成后浏览器保持打开，等你勾选完点发送，复用同一个 tab。

securityId 贯穿全链路：all_scored 数组里每个元素 8 个字段，最后一个是 si（securityId）。经过详情重评、分层、record_manager 存储，一路跟到前端 JobCard 组件的 job.securityId——发送时前端把它传回 /api/send。

第四层 · AI 决策层 httpx → DeepSeek

三个独立模块，各调各的 DeepSeek API，各有一个共享 httpx.AsyncClient（连接池上限 50、keepalive 10 条）。惰性初始化——第一次调时才创建，之后复用。

matcher.py — JD 匹配打分

输入：你的简历全文（前 3000 字）+ 职位描述（前 2000 字）→ DeepSeek 返回 0-100 分 + 匹配理由 + 置信度 + 是否虚假招聘 + 是否跨行业友好。Prompt 里写了评分标准：技术栈 50%、项目经验 30%、基本条件 20%、跨行业加分 5 分。

search_generator.py — 动态搜索词生成

输入：你的简历全文 → DeepSeek 返回 3-5 个搜索方向（方向名 + 关键词数组 + 优先级 + 推荐理由）。比如从你的简历里分析出「AI应用开发」「全栈开发」「LLM工程化」「AI辅助编程」四个方向。结合 config.yaml 里的城市和关键词做约束。

greeting_generator.py — 招呼语生成

输入：你的简历摘要 + 职位描述 → DeepSeek 返回 50-100 字的招呼语。Prompt 要求诚实——只能说你简历里真实存在的技能，"不要夸大年限、不要编造头衔"。

为什么选 DeepSeek：国内最便宜、中文最好、API 兼容 OpenAI 格式。三个模块用同一个 deepseek-chat 模型，只是 temperature 和 max_tokens 不同（打分 0.1、搜索词 0.3、招呼语 0.5）。

第五层 · 数据存储 本地 JSON 文件

config.yaml 平台设置（BOSS）、搜索城市/关键词/排除词、匹配分层参数、虚假招聘特征词、安全延迟、简历路径。所有模块通过 config_loader.py 统一读。
.env DEEPSEEK_API_KEY。.gitignore 排除。
my_resume.pdf 你的简历 PDF。config.yaml 里 resume.pdf_path 指向它。.gitignore 排除 *.pdf。
data/applied_jobs.json 已投递（干跑）记录。每条含公司、职位、分数、分层、理由、securityId、招呼语、时间戳。原子写入（写 .tmp 再 rename）。

两个问题

1. URL 可能不对：securityId 是 BOSS API 返回的字段，但职位详情页 URL 用的可能是 encryptJobId——搜索 API 返回里两个字段都有，我们不确定哪个是详情页 URL 的参数。

2. 沟通按钮不在详情页：GitHub 上 7 个同类项目的做法一致——在搜索列表页的职位卡片上直接找沟通按钮（文本可能是「立即沟通」「发起沟通」「沟通一下」「聊一聊」「联系Ta」「开始沟通」六种之一），点开后弹出一个招呼语对话框，填好发送。不跳详情页。

下一步需要的验证（你手动做，不改代码）：
① 打开 BOSS 网页版，随便看一个职位详情页，看浏览器地址栏 URL 格式是不是 job_detail/xxx.html
② 详情页上有「立即沟通」按钮吗？打开控制台执行 document.querySelector('[class*="chat"]') 看能找到什么
③ 回到搜索列表页，找一个职位卡片，右键检查元素，看沟通按钮的 class 和 DOM 结构

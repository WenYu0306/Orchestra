# JobHunter 开发日志 — 从脚本到 Agent 的 15 天

2026 年 7 月 15 日，李文禹

## 为什么做这个项目

我是建筑工程管理出身，转行 AI 开发。找工作这件事，我想用 AI 来帮忙。

BOSS 直聘上每天有成千上万个岗位，手动翻页、手动打分、手动写招呼语——整个过程慢、重复、消耗意志力。HR 端可以用 AI 自动筛选候选人，为什么求职者不能用 AI 来找工作？这不公平。

所以我决定做一个 Agent，帮我自动完成这些事。

## 项目路线——三次死胡同、三次翻盘

### 第一阶段：数据怎么拿？（6/30-7/2）

最初的想法跟大多数人不一样——不是在页面上爬 DOM，而是让 AI "看图点按钮"。我的第一个方案是接入了 Qwen VL 视觉大模型——截图当前页面，让视觉模型识别按钮位置，计算鼠标坐标，模拟轨迹点击，再截图看结果。这是一个"AI 看屏操"的套路。

跑了几次就发现了三个硬伤：一是视觉模型的识别速度跟不上页面变化，等它判断完"这个按钮在 x=150,y=340"，页面已经往下滚了一段；二是 BOSS 的页面结构复杂，按钮和对话框挤在一起，视觉模型常误判；三是这种模式的运行成本太高了点，每步都要截一张图发给 Qwen VL，一步一调用，全程下来调了几十次。

但 Qwen VL 的方案给我们带来了最关键的原始库——**nodriver**。这是 ultrafunkamsterdam 一个人开发的 Python 库，star 数不多，中文教程为零。它的核心设计思路是用 CDP（Chrome DevTools Protocol）直连 Chrome，不需要 WebDriver、不需要 Selenium。这个库在 2026 年的 GitHub 上几乎无人知晓，但它唯一的长处正是我们需要的——能绕过 BOSS 直聘的反爬检测。选择它是一场赌博，赌对了。

第二条路是传统的 DOM 爬取——用浏览器自动化翻搜索结果页，爬 DOM，拿职位列表。但也跑不通——DOM 爬取慢，而且只能看到页面渲染出来的内容，深层数据拿不到。搜索"AI应用开发"返回 30 个岗位，翻到第 5 页 BOSS 就开始给你看"相关推荐"而不是真实搜索结果。

两条路都堵了。第三条路是在 Chrome 页面里跑了一行同步 XHR：

```javascript
var x = new XMLHttpRequest();
x.open('GET', '/wapi/zpgeek/search/joblist.json?query=AI应用开发&city=101010100&page=1&pageSize=30', false);
x.send();
```

这行代码直接调了 BOSS 的内部搜索 API。数据是完整的 JSON，不需要跑 DOM 爬取，不需要翻页，不需要担心"第 5 页之后是假的"。这一步决定了我之后所有数据走的路线——不是跟别人一样爬 DOM，是调内部 API。

这个决策节省了后续所有的反爬成本和数据质量问题。

### 第二阶段：怎么评分、怎么分层？（7/3-7/7）

数据拿到了——每个城市、每个关键词能拿到 30 个岗位。但 600 个原始岗位，哪些是真正匹配我的？

第一版方案是 DeepSeek 一次性评分。Prompt 把简历全文 + 岗位描述发给 DeepSeek，让它打 0-100 分。但很快发现一个岗位的搜索字段太短——只有职位名、薪资、技能标签——这些信息不够支持精准评分。

所以做了双层评分：
- **标签评分**——搜索返回的字段拼成一句话，DeepSeek 快速评价，筛掉分太低的
- **详情重评**——前 30 个最高分的，用另一个 API（`card.json`）拿完整 JD，让 DeepSeek 重新打分

两轮之后，85 分以上的岗位数量从 5 个提升到 12 个。分出了三层——高分（≥80）、中分（≥60）、可试试（<60）。

这时候项目还只是一个流水线——搜完评完，手动选几个发。

### 第三阶段：勾选和发送（7/8-7/12）

这段时间是项目最难的三天。前端复选框的 bug 卡了我整整三天。

现象很简单——页面上有复选框，你点它，没反应。按钮永远是灰的。我改了六种方案——Set 改 Array、事件链改 Props、splice 改全量替换——每一次都要重跑 20 分钟的评分才能验证一次。三天跑了二十多次测试。

最后发现根因不是事件链的问题——是 SSE 推送 `record` 事件时，前端没取 `securityId` 字段。你的勾选动作传的是 `undefined`，函数第一行就跳过了。

这个 bug 的排查过程教会我一件事——不要自己造轮子。最后用的是 Vue 3 标准的 `v-model="job._selected"`，一行就完成了之前自定义 div、click 事件链、prop 传递轮子加起来都没做到的事情。

发送环节同样绕了一大圈。

第一步试的是在详情页找"立即沟通"按钮，点击后在弹出的对话窗口里找输入框填文字。但 BOSS 的对话窗口是 React Portal 弹出来的——不在当前页面的 DOM 树里。花了几个小时换 JS dispatchEvent、nodriver 元素 API、键盘 API，全部失败。

转折点在从按钮 HTML 里发现了两个关键信息——`data-url="/wapi/zpgeek/friend/add.json"` 和 `redirect-url="/web/geek/chat?id=..."`。前者是 BOSS 内部建立沟通关系的 API，后者是聊天页的地址。

**`friend/add.json` 的发现是一层层试出来的。** 我们先用 `querySelector` 扫了页面上所有带 `data-url` 属性的按钮——发现点完"立即沟通"之后，按钮的 HTML 里藏着这个 API 端点。它做两件事：建立你与老板的沟通关系，同时自动发一条系统默认招呼语——"您好，我觉得这个岗位非常适合我..."。这条拦不住，是平台固定的。

但是发现这个 API 之后又踩了新坑——它只发系统预填好的招呼语，不看我们传进去的自定义文案。所以我们走第二步：取 `redirect-url` 跳到聊天页，在聊天页的输入框里 CDP 插入 DeepSeek 生成的文字，单独再发一条。这就是 HR 手机上会收到两条消息的原因。

**还有一个让人崩溃了两小时的系统级 bug。** CDP 输完文字，回车键也发了，日志显示完成，但手机上看不到消息。排查发现不是代码问题——macOS 对后台 Chrome 窗口有键盘事件节流。你把 Chrome 缩到最小，Enter 就是发不出去；你让它在桌面前台正常窗口，一切正常。这个 bug 教会我们一个原则：Chrome 窗口一定要开着、不要缩。

最终方案分两步：
1. XHR 调 `friend/add.json` 建立沟通关系（BOSS 自己会带上系统默认招呼语）
2. 取 `redirect-url` 跳转到聊天页，用 Chrome 内核协议 `Input.insertText` 把 DeepSeek 生成的招呼语写进输入框，`Input.dispatchKeyEvent` 模拟回车发送

HR 手机上会看到两条消息——一条系统默认的、一条 DeepSeek 写的自定义文案。全开源项目里，这是第一个公开做到这点的。

### 第四阶段：从 Workflow 到 Agent（7/13-7/14）

到 7 月 13 号，项目的"搜→评→分层→发送"全链路已经跑通。但有个问题一直没解决——项目是一个固定流水线，不是真正的 Agent。

每次启动，所有关键词全部搜一遍，搜完所有分层的名额。关键词顺序是写死的、每个城市搜多少也是写死的。如果这一批搜索质量不好，项目不会自己调整——它会走完全部步骤，产出一个不太好的结果。

Agent 的核心特性是**观察→决策→执行**循环。每次行动后看结果，根据结果决定下一步。

改造分两层：

第一层——把 `_run()` 拆成 8 个独立方法。之前的代码是整个搜索、评分、分层过程全挤在一个 415 行的方法里，没有任何一段能独立测试。拆分后每个方法有明确的输入输出——搜一批、评一次、分一层，互不干扰。

第二层——Agent 循环。搜索改成原子工具 `_search_one(keyword, city)`，一次只搜一个词一个城。每轮搜完，把当前状态（候选数量、分数分布、城市分布、剩余关键词）拼成文本发给 DeepSeek，让它决定——搜什么词、搜哪个城、还是停了进分层。

DeepSeek 自己做了这些决策（来自实际日志）：
```
DeepSeek: search LLM工程化/北京 — 北京AI岗位集中，LLM工程化匹配度更高
DeepSeek: search Python+React/北京 — 北京候选人多，优先搜索高匹配度
DeepSeek: 候选池达190，≥70分38个 → 停止搜索，进入分层
```

最后的引擎是两层——Agent 自主调度 + 硬规则兜底（300 候选或 12 轮即停）。Agent 负责选择最优策略，硬规则防止跑飞。

### 第五阶段：优化和稳定（7/14-7/15）

从"能跑通"到"能持续使用"大概花了一整天，修了十几个小问题：

- 招呼语温度调成 0.8，第一人称，不说客套话
- 分层从"按位置填坑"改成"按分数阈值"（≥80/≥60/<60）
- Chrome 主窗口必须开着，不能最小化（后台会节流键盘事件）
- 聊天输入框捕获——三层兜底（chat 容器 → textarea/div → 排除搜索框）
- CDP 回车补上 `char` 事件（React 需要 keyDown→char→keyUp 三连才认）
- 简历上传、一键启动脚本、零基础安装指南全部补齐

## 架构是怎么演化的

最终的三层架构不是设计出来的——是代码撞了墙重新长的。

**第一次：单文件。** 所有逻辑塞在 `agent_runner.py` 里——搜数据、调 DeepSeek、打分、分层、发消息。改一次崩一次，永远不知道哪里坏了。

**第二次：拆模块。** matcher、search_generator、greeting_generator 各自独立文件。每个模块自己管理 httpx 连接、自己处理异常。改打分不会影响搜索词生成，修招呼语不会崩掉评分。三层架构的雏形——连接层(nodriver)、执行层(httpx+DeepSeek)、决策层(agent_runner 编排)——就是这次拆分自然形成的。

**第三次：双层评分。** 一开始是"搜一个词→打一次分"，精准但也慢。200 个岗位评分花了太多时间，而且大部分岗位的搜索字段信息量极薄——只有职位名、薪资、技能标签，DeepSeek 拿这几行字也打不准。改成"标签快速初筛 + top 30 详情重评"之后，85 分以上的岗位从 5 个提升到 12 个。

**第四次：Workflow 改 Agent 循环。** 项目跑通了但有个死命题——所有关键词全部搜一遍，不管质量好不好。从 `for city: for kw:` 这种双重循环，改成了 `while` 循环——每轮只搜一个词一个城，搜完看状态，DeepSeek 决定下一步：搜什么词、搜哪个城、还是停。最后补了一条硬规则兜底——300 候选或 12 轮主动停。

**发送层：加了又撤、撤了又加。** 从 JS dispatchEvent → nodriver 元素 API → CDP `insert_text`。中间还砍过 `friend/add.json` 改成对话框 CDP 直接发，发不出去又回退。最终版本分两步——XHR 调 API 建关系（带系统默认语）+ 跳聊天页 CDP 发 DeepSeek 自定义语。稳定了。

整个架构的演化路径：单文件 → 模块化 → 双层评分 → Agent 循环 → 两层发送。每一步都是项目本身逼出来的——不是学了某个设计模式再套进去。

## 架构

```
决策层（Agent 自主调度）→ 执行层（httpx → DeepSeek）→ 连接层（XHR 获取数据 + CDP 发送消息）
```

## 最后的话

一个不会写代码的人，用了 13 天，从零做了一个全链路 AI Agent 求职系统。评分、分层、Agent 调度、CDP 自定义文案——每一步都是被问题推着走的，每一层架构都是自己撞墙撞出来的。它不是什么"产品级系统"，但它是一个完整的证据——证明 AI 不只是帮你写代码，它能帮你做你不敢开始的事。

如果你也在转行，也在找第一份 AI 岗位——希望这个项目告诉你：有些事，不需要你会写代码才敢做；你只要先开始，它就慢慢变成你可能完成。
